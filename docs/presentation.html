What does it do?

🔄

Multi-source data sync

Pulls transactions from pluggable data sources — mock banks, card providers, payment networks, and Plaid — through a unified port interface. No business logic changes when adding a new source.

📊

Aggregation & analytics

Groups transactions by account and time period (hourly → yearly). Computes per-category spend breakdowns with rule-based keyword classification.

⚡

Real-time Kafka ingestion

Batch consumer reads synthetic transaction events off Kafka topics, persists via reactive R2DBC pipeline, routes failures to a Dead-Letter Topic for safe reprocessing.

🔐

Secured REST API

JWT (HMAC-SHA256) with two roles — Customer and Admin. Rate-limited token issuance. Fine-grained access control: customers see only their own data.

Architecture — Hexagonal (Ports & Adapters)

Dependency Rule

Dependencies point inward only. The domain knows nothing about Spring, the database, or Kafka. Adapters implement ports defined by the domain.

Enforced by ArchUnit

Architecture rules are compiled into tests. A bad import fails the build. No discipline required at review time.

Why it matters

  • Swap a data source without touching business logic
  • Unit-test domain services with no DB or broker
  • Migrate from PostgreSQL to another DB by writing one adapter

Why reactive?

One event-loop thread handles thousands of concurrent connections without blocking. Critical for an aggregator that fans out to multiple upstream sources simultaneously.

Blocking rule

BlockHound is active in tests. Any blocking call on a reactor thread fails the test immediately. Flyway is the only exception — it runs before the reactor starts on a dedicated thread.

Key Architectural Decisions (ADRs)

ADR-001

Hexagonal Architecture

Domain logic is framework-independent. Adapters are mechanical — swappable without touching business rules. Enforced at build time by ArchUnit tests.

ADR-002

Reactive Stack (WebFlux + R2DBC)

Non-blocking I/O end-to-end. Backpressure propagates from DB all the way to the HTTP response. Required tradeoff: no blocking calls in the Reactor pipeline.

ADR-003 + 004

Self-Issued JWTs + Vault

App issues its own HMAC-SHA256 tokens for evaluation. Secrets sourced from HashiCorp Vault in docker/prod. Local dev fallback with obvious keys prevents accidental production use.

ADR-005

Kafka Schema Versioning

Explicit version field in event payloads. Consumer tolerates unknown fields. Supports rolling upgrades with consumer lag — old consumers continue processing until restarted.

ADR-006

API Versioning Policy

URL path versioning at /api/v1/. Explicit definition of breaking vs. non-breaking changes. Deprecated versions supported 6+ months; sunset returns 410 Gone.

Design Choice

Rule-Based Categorisation

Keyword matching on merchant name + description. Ordered rules — TRANSFER before FOOD prevents "Uber Eats" being classified as TRANSPORT. Returns OTHER if no rule matches.

Pluggable Sources

Each data source implements FetchTransactionsPort. Adding a new provider means writing one adapter — zero changes to business logic.

Fault Isolation

Errors in one source are logged and counted but do not cancel the others. All sources run independently via reactive Flux chains.

Idempotent Upserts

Re-syncing the same date range is safe. Transactions are upserted by ID — duplicates update in place rather than creating new rows.

Batch Listener

Consumes a list of records per poll rather than one at a time. Reduces broker round-trips and amortises the R2DBC connection overhead across up to 250 events per batch.

Manual Offset Commit

The consumer holds the offset until the full reactive pipeline completes. If the JVM crashes mid-batch, Kafka redelivers the entire batch on the next poll — no events are silently lost.

Dead-Letter Topic

Events that exhaust all retries are published to a DLT instead of blocking the main topic. Operations can inspect, replay, or discard them independently without stalling healthy consumers.

Exactly-Once Guarantee

  • R2DBC upsert by ID — re-processing the same event is idempotent
  • Offset only committed after DB write succeeds
  • Combined: at-least-once delivery + idempotent writes = effectively exactly-once
Retry strategy:
Attempt 1 → Wait 200 ms → Attempt 2 → Wait 200 ms → Attempt 3 → Route to DLT
Offset NOT committed on DLT route → broker will not redeliver

Transaction Categorisation Engine

Ordered Rule Chain — first match wins

1
SALARY salary · payroll · monthly pay
2
RENT_AND_MORTGAGE rent · mortgage · landlord
3
TRANSFER payment to · wire transfer · transfer
↑ must come before FOOD — "Uber Eats payment to..." would match TRANSPORT otherwise
4
FOOD_AND_DINING restaurant · uber eats · deliveroo
↑ must come before TRANSPORT — "Uber Eats" shares keyword with Uber ride
5
TRANSPORT uber · train · bus · taxi · fuel
…
UTILITIES · SHOPPING · HEALTHCARE · EDUCATION · INSURANCE
∅
No match → OTHER

Worked Example

input: "Uber Eats order #1234"
rule 1 SALARY → no match
rule 2 RENT → no match
rule 3 TRANSFER → no match
rule 4 FOOD → "uber eats" ✓ match
result: FOOD_AND_DINING

Pure Domain Logic

The categorisation service has zero framework imports. It takes a string, walks an ordered list of rules, and returns an enum. Trivially unit-testable and swappable — a future ML-based classifier just needs to implement the same port.

Order is a contract

Changing rule order is a breaking change — it silently reclassifies existing transactions on re-sync. Any reordering requires a migration or a versioned ruleset.
